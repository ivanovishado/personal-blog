La mayoria de los sistemas distribuidos fallan no por malos algoritmos, sino por el acoplamiento fuerte entre servicios que se suponia debian ser independientes. La arquitectura event-driven aborda esto a nivel estructural: los servicios se comunican a traves de eventos en lugar de llamadas directas, lo que significa que cada servicio puede evolucionar, escalar y fallar de forma independiente. Despues de construir sistemas event-driven en fintech, logistica y e-commerce, he encontrado que la brecha entre la teoria y produccion es donde la mayoria de los equipos tienen dificultades.

Este post cubre los patrones que realmente funcionan, las trampas que parecen buenas ideas, y los modelos mentales que hacen que la consistencia eventual se sienta natural en lugar de aterradora.

## Por que los Eventos Superan a Request-Response a Escala

Una encuesta de 2024 de la Cloud Native Computing Foundation encontro que el 78% de las organizaciones que ejecutan mas de 50 microservicios adoptaron alguna forma de comunicacion event-driven. La razon es directa: request-response crea dependencias invisibles en tiempo de ejecucion. Cuando el Servicio A llama al Servicio B de forma sincrona, A no puede funcionar si B esta caido. Multiplica esto por docenas de servicios y obtienes fallos en cascada que tumban plataformas enteras un sabado por la noche.

Los eventos invierten esta relacion. Un productor emite un hecho sobre algo que ocurrio -- "orden fue colocada," "pago fue capturado," "inventario fue reservado" -- y se desentiende. Cualquier numero de consumidores puede reaccionar a ese evento en su propio tiempo.

Aqui hay un productor de eventos minimo usando TypeScript y una abstraccion de message broker:

```typescript
interface DomainEvent<T = unknown> {
  id: string;
  type: string;
  occurredAt: Date;
  aggregateId: string;
  payload: T;
}

interface OrderPlacedPayload {
  orderId: string;
  customerId: string;
  items: Array<{ sku: string; quantity: number; price: number }>;
  total: number;
}

async function placeOrder(
  command: PlaceOrderCommand,
  broker: EventBroker,
): Promise<Order> {
  const order = Order.create(command);

  const event: DomainEvent<OrderPlacedPayload> = {
    id: crypto.randomUUID(),
    type: 'order.placed',
    occurredAt: new Date(),
    aggregateId: order.id,
    payload: {
      orderId: order.id,
      customerId: command.customerId,
      items: command.items,
      total: order.total,
    },
  };

  await broker.publish('orders', event);
  return order;
}
```

La idea clave: la funcion `placeOrder` no sabe ni le importa quien consume el evento. El servicio de pagos, el servicio de inventario y el servicio de notificaciones se suscriben de forma independiente. Agregar un nuevo consumidor de analiticas requiere cero cambios en el servicio de ordenes.

## Event Sourcing: Cuando Necesitas la Historia Completa

Event sourcing lleva la idea event-driven mas alla: en lugar de almacenar solo el estado actual de una entidad, almacenas cada evento que llevo a ese estado. Esto te da un registro de auditoria completo, la capacidad de reconstruir el estado en cualquier punto del tiempo, y la opcion de construir nuevos modelos de lectura a partir de datos historicos.

El tradeoff es real. Los sistemas con event sourcing requieren mas almacenamiento, un diseno de esquemas de eventos mas cuidadoso, y un enfoque diferente para las consultas. No puedes simplemente hacer `SELECT * FROM orders WHERE status = 'shipped'` -- necesitas un modelo de lectura proyectado.

Aqui hay un patron simplificado de event store y proyeccion:

```typescript
interface EventStore {
  append(streamId: string, events: DomainEvent[]): Promise<void>;
  read(streamId: string, fromVersion?: number): Promise<DomainEvent[]>;
}

// Reconstruir el estado actual de una orden desde su stream de eventos
function rebuildOrder(events: DomainEvent[]): OrderState {
  return events.reduce<OrderState>((state, event) => {
    switch (event.type) {
      case 'order.placed':
        return {
          ...state,
          id: event.payload.orderId,
          status: 'placed',
          items: event.payload.items,
          total: event.payload.total,
        };
      case 'order.confirmed':
        return { ...state, status: 'confirmed' };
      case 'order.shipped':
        return {
          ...state,
          status: 'shipped',
          trackingNumber: event.payload.trackingNumber,
        };
      case 'order.cancelled':
        return { ...state, status: 'cancelled', reason: event.payload.reason };
      default:
        return state;
    }
  }, {} as OrderState);
}
```

Este patron brilla en dominios donde "que paso y cuando" importa tanto como "cual es el estado actual." Transacciones financieras, flujos de trabajo con requerimientos de cumplimiento, y edicion colaborativa son casos naturales. Una app CRUD con transiciones de estado simples probablemente no lo necesita -- y agregarlo prematuramente hara que tu equipo avance mas lento.

## CQRS: Separando Lecturas de Escrituras

Command Query Responsibility Segregation (CQRS) se combina naturalmente con event sourcing. El lado de escritura maneja comandos y produce eventos. El lado de lectura se suscribe a esos eventos y mantiene modelos de consulta optimizados. Cada lado puede usar el modelo de datos y el motor de almacenamiento que mejor le convenga.

En la practica, esto significa que tu modelo de escritura podria ser un stream de eventos en Kafka o EventStoreDB, mientras que tu modelo de lectura vive en PostgreSQL con tablas desnormalizadas afinadas para las consultas exactas que tu UI necesita.

```typescript
// Lado de escritura: manejar un comando, producir eventos
async function handleShipOrder(
  command: ShipOrderCommand,
  eventStore: EventStore,
): Promise<void> {
  const events = await eventStore.read(command.orderId);
  const order = rebuildOrder(events);

  if (order.status !== 'confirmed') {
    throw new Error(`Cannot ship order in "${order.status}" status`);
  }

  await eventStore.append(command.orderId, [
    {
      id: crypto.randomUUID(),
      type: 'order.shipped',
      occurredAt: new Date(),
      aggregateId: command.orderId,
      payload: { trackingNumber: command.trackingNumber },
    },
  ]);
}

// Lado de lectura: proyectar eventos en un modelo consultable
async function projectOrderShipped(
  event: DomainEvent,
  db: Database,
): Promise<void> {
  await db.query(
    `UPDATE order_read_model
     SET status = 'shipped',
         tracking_number = $1,
         updated_at = $2
     WHERE order_id = $3`,
    [event.payload.trackingNumber, event.occurredAt, event.aggregateId],
  );
}
```

Lo critico que los equipos hacen mal con CQRS es aplicarlo en todas partes. Si tu servicio tiene operaciones CRUD simples sin patrones de consulta complejos, un modelo compartido esta bien. CQRS agrega complejidad operacional -- dos almacenes de datos que monitorear, consistencia eventual entre ellos, y lag de proyeccion que considerar en la UI. Usalo donde los patrones de lectura y escritura genuinamente divergen.

## Abrazando la Consistencia Eventual

La parte mas dificil de los sistemas event-driven no es el codigo -- es el cambio mental. En un mundo sincrono, cuando guardas datos y luego los lees, esperas ver tus cambios. En un mundo event-driven, hay una ventana donde la escritura ocurrio pero el modelo de lectura aun no se ha actualizado.

Tres estrategias que hacen esto manejable:

1. **Actualizaciones optimistas en la UI.** Despues de una accion del usuario, actualiza la UI inmediatamente con el resultado esperado mientras el evento se propaga en segundo plano. Si la operacion falla, revierte el estado de la UI.

2. **Consistencia causal donde importa.** Usa numeros de version o relojes vectoriales en las rutas criticas. Cuando el usuario que acaba de colocar una orden ve su historial de ordenes, enruta esa consulta especifica a un almacen garantizado para incluir su ultimo evento.

3. **Consumidores idempotentes.** Los eventos se entregaran mas de una vez. Disena cada consumidor para manejar duplicados de forma elegante. Almacena el ID del evento y omite el procesamiento si ya lo has visto antes:

```typescript
async function handleEvent(
  event: DomainEvent,
  db: Database,
): Promise<void> {
  const processed = await db.query(
    'SELECT 1 FROM processed_events WHERE event_id = $1',
    [event.id],
  );

  if (processed.rows.length > 0) {
    return; // Ya manejado, omitir
  }

  await db.transaction(async (tx) => {
    await processBusinessLogic(event, tx);
    await tx.query(
      'INSERT INTO processed_events (event_id, processed_at) VALUES ($1, $2)',
      [event.id, new Date()],
    );
  });
}
```

## Por Donde Empezar

Si estas considerando la arquitectura event-driven, empieza pequeno. Elige un limite en tu sistema donde dos servicios estan fuertemente acoplados a traves de llamadas sincronas. Reemplaza ese punto de integracion con un evento: el servicio productor emite un evento, y el servicio consumidor se suscribe.

Notaras tres cosas de inmediato: los deploys se vuelven independientes, los fallos parciales dejan de propagarse en cascada, y agregar nuevos consumidores se vuelve trivial. Una vez que hayas sentido esos beneficios de primera mano, el camino hacia event sourcing y CQRS -- si y cuando los necesites -- se vuelve mucho mas claro.

El objetivo no es construir el sistema event-driven mas sofisticado posible. El objetivo es construir un sistema donde los componentes puedan cambiar de forma independiente, fallar de manera elegante, y escalar en sus propios terminos. Los eventos son el mecanismo. El acoplamiento debil es el punto.
